Griffin

Griffin is a homebrew 68000 computer I started building to use up some old chips. The Grantham family crest features a griffin, which I had forgotten, so the name turned out to be very appropriate. The source and design files are on GitHub.


The board on the bench, May 2026.

The starting point was the parts drawer. I had a 12 MHz 68000, a couple of 128K AS6C4008-55 SRAMs, some 64K EPROMs that need to be erased with 20 minutes under an ultraviolet lamp, and the usual assortment of LEDs, transistors, capacitors, and resistors that accumulates over forty years. I bought a few new parts to fill in the gaps: an MC68681 DUART, three ATF1508AS programmable logic devices, and EPROMs that could be erased directly in a USB programmer. A 68010 is on the way in case I want to try Linux NOMMU later.

The goal was modest. Boot the CPU, talk to a serial console, read a CF card, take input from a PS/2 keyboard, and eventually display a bitmap on a screen. Audio would be a bonus. None of this is groundbreaking — people have built dozens of 68k boards — but I wanted one made out of parts I already had.

The design lives in YAML

Early on I decided that the machine definition would live in a single file, griffin.yml. It contains the register addresses, bitfield definitions, queue sizes, clock constants, and the protocols between peripherals. A top-level make generates C++, Verilog, and assembly headers from it, so the firmware, the CPLD logic, and the emulator all agree on what bit means what.

There's also griffin.md, which is more of a running design journal. Looking back through the git history of those two files, the project changed its mind a lot. Composite video got added, deleted, and partly added back. The serial part went from an 8051-style UART to the MC68681. An MMU appeared and then went away. A "rev 2" rethink started before rev 1 was even fully working. Most of the meandering happened because something didn't fit in the CPLD, or because I changed my mind about what I cared about.

Where it is now

The rev 1 board boots at an overclocked 14 MHz. It has:

  • A 68000 CPU (probably a 68010 later).
  • 1 MB of SRAM, two AS6C4008-55s in a 16-bit pair.
  • Two 27C512 EPROMs for ROM, even and odd.
  • An MC68681 DUART for the serial console and the system tick.
  • A CF card interface with FAT support, via ElmChan's FatFs.
  • PS/2 keyboard input, mostly working.
  • A CPU-writable audio DAC with no hardware assistance.
  • 640×480 1bpp VGA out of main memory, with two R3G3B2 palette entries the CPU can rewrite every frame.

The console comes up, the ROM monitor talks, the CF card enumerates and FAT reads work, and PS/2 is good enough that I can print keycodes. Video is now up: VIDEO DMAs a 640×480 1bpp framebuffer and drives VGA, picking each pixel from one of two palette slots. The slots hold R3G3B2 values and are CPU-writable, so even though there are only two colors on screen at once, the firmware can change which two between frames in the vertical-blank ISR — enough for a splash screen with a flash, or for crude palette-cycling effects.

The rev 1 design files, which have yet to be updated with swapping out the AT89S52 for an XR68C681 with the video design, are in the repo: the schematic, the PCB, and a distilled netlist summarized from KiCad.

CPLDs and the CPU

The most interesting decisions on this project have been about where to draw the line between the CPU and the three ATF1508 CPLDs, which I named GLUE, VIDEO, and ENGINE. The CPLDs are good at deterministic, parallel, fast signaling, but their real estate is tiny — on the order of 128 macrocells each. The CPU is flexible but it stalls on DTACK, takes interrupts, and waits on slow ROM, so it's not great for hard real-time work.

A full UART would not fit in GLUE, so instead GLUE just provides a free-running reloadable timer. The CPU does the actual UART RX and TX in a tight polled loop with deterministic ticks. That worked well enough for bringup, and it has since been replaced by the MC68681's hardware UART. The trick still earned its keep.

PS/2 input is split the same way. GLUE synchronizes the PS2_CLK and PS2_DATA lines, detects each falling edge of the clock, latches the data bit, and raises an IRQ — the bit-level work that the CPU has no business trying to do at that timing. The CPU then handles the framing on top of that: start/stop/parity bits, packing into bytes, and scancode decoding. Bit-level timing in the CPLD, byte-level protocol in firmware.

The CPLD toolchain

The ATF1508 toolchain is its own small adventure. I am driving Verilog through synthesis into EDIF, and then through Atmel's old fit1508 binary. A Python helper summarizes the fit report and checks that the pin assignments have not drifted. The pins on the rev 1 board are already routed, so the fitter has to be told to keep the pinout exactly. The Makefile in cpld/ wraps all of that, and the resulting .svf files get programmed in over JTAG.

The emulator

There's a small 68000 emulator in emulator/ that models the CPU and the MMIO peripherals well enough to run the ROM. It has turned out to be very useful: I can iterate on firmware without pulling the EPROMs, which early on meant erasing them under UV, and reflashing, which is still a real time sink. It also serves as a sanity check — if a change behaves the same on the emulator and the real board, I have more confidence the bug is somewhere I understand, and if they disagree, that disagreement is itself a useful clue.

What's next

With VGA working, next is a proper keyboard-and-screen console, and then some kind of programming environment like BASIC. I'd like FUZIX or ucLinux if I can get those to work.

A note on Claude Code

I have leaned heavily on Claude Code throughout the project, both for typing boilerplate and as a design consultant I can argue with at midnight. It has helped me untangle CPLD fitter errors, draft emulator code, write firmware in C++23, and generally keep momentum on a project I would otherwise have shelved several times. A lot of the cleaner code on this project would not exist without it, and probably some of the bugs would not either.
